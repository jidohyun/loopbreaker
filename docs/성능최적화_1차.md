# 모델 성능 최적화 포트폴리오 (1차) — LoopBreaker

> Baseline 모델 구현 및 초기 성능 분석. 모든 Baseline 성능은 실제 실험(실 세션 골드셋 72건 리플레이) 결과 기반.

## 1. 기본 정보

- **프로젝트 제목:** LoopBreaker — 코딩 AI 에이전트의 의미적 실패(헛돌기·가짜 성공) 탐지 모델
- **학번:** 3413
- **이름:** 지도현

## 2. 문제 정의 및 예측 목표

- **해결하려는 문제:** 자율 AI 코딩 에이전트(Claude Code 등)가 빠지는 **의미적 실패**를 자동 탐지하는 것. ① thrashing(같은 파일을 미세하게 바꿔가며 반복 편집하는 헛돌기) ② false_success(근거 없이 "완료"를 선언하거나 자기참조로 검증을 통과시키는 가짜 성공).
- **문제 발생 배경:** 토큰 수·요청 수 같은 결정론적 규칙으로는 이 실패가 안 잡힌다(입력이 조금씩 달라지며 규칙을 빠져나가고, "완료 선언"은 형식상 정상). 의미를 읽어야만 잡힌다. 실제로 Replit·Amazon Kiro의 프로덕션 사고가 발생했고, 본인 dohyun 하니스에서도 자기승인 우회 5건이 실측됐다.
- **모델의 예측 목표:** 세션 로그 윈도가 thrashing / false_success / none(정상) 중 어디에 해당하는지 **다중 클래스 분류**.
- **입력 데이터:** Claude Code 세션 JSONL을 정규화·정렬한 이벤트 시퀀스(ActionTriple), 그리고 슬라이딩 윈도 단위의 구조 신호(반복 편집 횟수, 동일 영역 Jaccard 유사도).
- **출력 결과:** 각 윈도/앵커에 대한 탐지 라벨(thrashing/false_success/none)과 근거(증거 이벤트·신뢰도). 평가 시 골드 라벨과 매칭해 혼동행렬 산출.

## 3. 데이터셋 설명

- **데이터 이름:** LoopBreaker 골드셋(`.goldset/goldset.eval.db`)
- **데이터 출처:** 본인의 실 Claude Code 세션 로그(`~/.claude/projects/**/*.jsonl`) 1,845개 마이닝 + dohyun 하니스 자기승인 우회 기록(합성 적응).
- **데이터 개수(전체 / Train / Test):** 전체 **72건**. 별도 가중치 학습이 없는 룰+임계 모델이라 Train/Test를 고정 분할하지 않고, 캘리브레이션 단계에서 stratified k-fold(k=5)로 내부 분할. 본 1차 Baseline은 **전체 72건 리플레이(Test 전수)** 기준으로 측정.
- **데이터 특징:** thrashing 31(실 JSONL)·none 36(실 JSONL)·false_success 5(dohyun 합성). 클래스 불균형(43% / 50% / 7%). 라벨은 휴리스틱 약라벨(`auto_heuristic`, 사람 라벨 아님). false_success는 실 JSONL 양성이 0건이라 합성으로만 구성.

## 4. 데이터 전처리 과정

- **결측치 처리 방법:** 파싱 실패 라인은 건너뛰고 전체 중단 금지. sessionId가 비면 파일명으로 폴백.
- **데이터 정규화 방법:** 라인→NormalizedEvent(top-level type, tool/input 추출), ts→parentUuid 위상→byteOffset 순 정렬. **(1차 Baseline에서는 평가 매칭 시 gold span을 연속 전체 이벤트 슬라이스 그대로 사용 — 정규화 미적용.)**
- **데이터 분할 방법:** 캘리브레이션 stratified k-fold(k=5). 소표본(클래스당 <30)이면 grid 축소 + 정성폴백.
- **데이터 증강 / Feature Engineering:** ActionTriple(actor/verb/object) 추출, 슬라이딩 윈도 반복 카운트(structuralRepeatCount), 동일 영역 Jaccard 유사도, false_success는 immediateApproval(Δts≈0)·circularRef(순환참조) 패턴으로 합성.

## 5. Baseline 모델 설계

- **사용 알고리즘:** 2단계 하이브리드 탐지. ① 구조 게이트(룰 기반: 슬라이딩 윈도 반복 편집 + Jaccard 미세변형 판정) ② 의미 판정(임베딩 코사인 유사도 + LLM-as-judge). 평가 매칭은 thrashing=IoU span(τ=0.5), false_success=anchor±k(k=5).
- **모델 구조:** events → 구조 게이트(DetectionHit) → hits→triples → 의미판정 STAGE2(embed) → judge → DetectionRecord. 평가 단계에서 예측 윈도(gate.windowRefs)와 골드 라벨(span/anchor)을 매칭 규칙으로 비교해 혼동행렬 산출.
- **주요 하이퍼파라미터:** fileEditWarn/fileEditCrit(반복 임계), simThresh=0.9, Jaccard 0.3(미세변형), 평가 IoU τ=0.5, anchor k=5, minSupport=15(소표본 정성폴백 경계).
- **학습 방법:** 가중치 학습 없음(룰+임계). 본 1차 Baseline은 **임계 기본값(DEFAULT_DETECTOR_CONFIG)** 으로 골드셋 전수 리플레이. 임베딩·judge는 결정론 Mock(부수효과 0, 재현 가능).

## 6. 성능 평가 방법

| 평가 지표 | 사용 여부 | 설명 |
|---|---|---|
| Accuracy | △ (부록) | 클래스 불균형이라 주 지표 아님. 부록 참고용. |
| F1-score | ✅ 사용 | 클래스별 F1 + Macro F1(불균형 보정 주 지표) + Micro F1. |
| RMSE | ❌ 미사용 | 회귀 지표 — 본 분류 과제엔 해당 없음. |

> 추가로 Precision/Recall(클래스별), Balanced Accuracy, Cohen's κ, Wilson 95% CI를 사용. 소표본 클래스(false_success, support 5<15)는 F1/κ를 생략하고 정성폴백.

## 7. Baseline 모델 성능 결과

| 실험 | 모델 | 평가지표 | 성능 |
|---|---|---|---|
| Baseline | 2단계 탐지 + 기본 임계(정규화 미적용) | thrashing P/R/F1 | 0.000 / 0.000 / 0.000 (TP=0, FP=59, FN=31) |
| Baseline | 〃 | none P/R/F1 | 0.500 / 0.379 / 0.431 |
| Baseline | 〃 | false_success | 정성폴백(support 5 < 15) |
| Baseline | 〃 | **Macro F1 / Micro F1** | **0.216 / 0.275** |
| Baseline (다른 모델) | "전부 none으로 예측"(다수 클래스) | Macro F1 | 약 0.22 (none만 맞춤, 소수 클래스 0) |

> Baseline의 thrashing F1이 0인 것은 모델이 헛돌기를 전혀 못 잡아서가 아니라, **평가 매칭 단위 불일치** 때문이다(8장 원인 분석 참조). 구조 게이트 자체는 thrashing 후보 윈도를 다수 발화했다(FP=59).

## 8. 초기 모델 분석

- **모델의 장점:** 룰 기반 구조 게이트가 thrashing 후보를 풍부하게 포착(높은 재현 후보). 의미판정·judge를 인터페이스+Mock으로 추상화해 부수효과 0, 결정론 재현 가능. 실 세션 기반 골드셋으로 측정.
- **모델의 한계:** ① 평가 단계에서 gold span(연속 전체 이벤트, 예 117개)과 pred 윈도(구조 게이트 file_edit만, 예 13개)의 **단위가 달라** IoU가 항상 ≈0.11로 떨어져 동일 구간도 매칭 실패. ② false_success는 실 양성 데이터가 없어 합성 5건뿐(소표본). ③ 단일 라벨러·휴리스틱 약라벨.
- **성능이 낮은 원인 분석:** Macro F1 0.216의 직접 원인은 **thrashing F1=0**이고, 그 원인은 학습 부족이 아니라 **평가 매칭의 단위 불일치 버그**다. gold span을 전체 이벤트로, pred 윈도를 file_edit로 정의해 둘의 IoU가 |file_edit|/|전체| 수준으로 희석됐다. 즉 모델은 같은 구간을 가리키는데 채점기가 다른 단위로 비교해 전부 FN 처리했다.
- **개선이 필요한 부분:** ① 평가 매칭에서 gold span과 pred 윈도를 **같은 구조신호(file_edit) 단위로 정규화**. ② false_success 실 양성 데이터 확보. ③ 임계 캘리브레이션으로 FP 억제.

## 9. 향후 성능 개선 계획

다음 단계(2차)에서 시도할 방법:

- **평가 매칭 정규화(최우선):** `matchThrashing`에 structuralUuids(pred windowRefs 합집합 ∪ gold span 양끝)를 도입해 gold span을 file_edit 단위로 축소 → 동일 구간 IoU≈1.0으로 회복. 부분겹침은 gold 끝이 union에 남아 FN 유지(과탐 방지).
- **하이퍼파라미터 조정:** simThresh / fileEditCrit / decideThresh를 stratified k-fold 격자 탐색(objective=macroF1−λ·FPR_none, λ=0.5)으로 캘리브레이션.
- **데이터 증강:** false_success 합성 패턴 확대, 다중 라벨러 라벨링으로 intra-rater κ 천장 상향.
- **추가 검증:** nested cross-validation으로 캘리브레이션 과적합 방지, 실 Anthropic judge API record 모드로 의미판정 정량치 측정.
