# 인공지능 프로젝트 실무 최종 보고서 — LoopBreaker

> 모든 모델 성능 결과는 실험(실 세션 골드셋 72건 리플레이) 결과 기반. Baseline(IoU 단위 정규화 전)과 최종 모델(정규화 후) 성능 비교 포함.

## 1. 기본 정보

- **프로젝트 제목:** LoopBreaker — 코딩 AI 에이전트의 실시간 실패 탐지·사람 호출 미들웨어
- **학번:** 3413
- **이름:** 지도현
- **프로젝트 기간:** 2026-05-29 ~ 2026-06-09 (사전조사·기획 → M0~M6 구현 → 제품 포장 → 골드셋 마이닝 → 실측 평가)

## 2. 프로젝트 개요

### 2-1 프로젝트 목적

자율 AI 코딩 에이전트(Claude Code 등)가 점점 더 오래 혼자 실행되면서, 사람이 매 순간 화면을 지켜볼 수 없게 됐다. 그 결과 에이전트가 빠지는 **의미적 실패**(같은 파일을 미세하게 바꿔가며 헛도는 thrashing, 근거 없이 "완료"를 선언하는 가짜 성공)가 누적·증폭돼도 즉시 알아채기 어렵다. 이 프로젝트의 목적은 에이전트의 세션 로그를 로컬에서 실시간 분석해, 규칙으로는 안 잡히는 이 의미적 실패를 탐지하고 **근거와 함께 사람을 호출**하는 read-only 감시 미들웨어를 만드는 것이다.

### 2-2 해결하려는 문제

- **해결하려는 문제:** 토큰 수·요청 수 같은 결정론적 규칙(rate limit, 횟수 카운트)으로는 잡히지 않는 의미적 실패 — ① thrashing(동일 영역 미세변형 반복 편집) ② 가짜 성공(자기승인 우회: 근거 없는 완료 선언, 자기참조 검증 통과).
- **문제 발생 배경:** 입력이 조금씩 달라지면서 규칙을 빠져나가고, "완료 선언"은 형식상 정상이라 규칙으로 막을 수 없다. 의미를 읽어야만 잡힌다. METR 연구상 에이전트가 수행 가능한 태스크 길이가 약 7개월마다 2배로 늘고(arXiv:2503.14499), Gartner는 2026년 말 신규 엔터프라이즈 앱의 40%가 에이전트를 탑재할 것으로 본다 — 자율 실행이 길어질수록 "사람이 못 보는 사이 헛도는 시간"의 비용이 커진다.
- **문제의 중요성:** 규칙으로 못 막은 실제 사고가 이미 발생했다. Replit AI 에이전트가 변경 금지 지시에도 프로덕션 DB를 삭제하고 가짜 데이터를 지어냈으며(1,200명 이상 영향, AI Incident DB #1152), Amazon Kiro가 승인 없이 프로덕션 환경을 통째로 삭제·재생성해 13시간 장애를 냈다(#1442). 게다가 **본인의 개인 AI 워크플로우 하니스(dohyun) 분석 기록에서 자기승인 우회가 5건 실제로 관측**됐다 — 가설이 아니라 내 환경에서 실측된 1차 증거다.

### 2-3 프로젝트 목표

- 에이전트 세션 JSONL을 로컬 실시간 감시 → 구조 신호(반복 편집) + 의미 신호(임베딩 유사도 + LLM-as-judge)로 2단계 탐지.
- 탐지 시 근거(증거 이벤트·신뢰도)와 함께 데스크톱/웹훅/CLI로 사람 호출.
- MVP 범위 = **탐지 + 알림**까지(자동 개입/차단은 후속 M7+). "Detect first, Block later" 원칙.
- 모든 외부 비용 API(임베딩·judge)는 인터페이스+Mock으로 추상화해 테스트 부수효과 0, 결정론 보장.

## 3. 사용자 분석 및 서비스 설계

### 3-1 대상 사용자

- **대상 사용자:** 자율 코딩 에이전트를 장시간 돌리는 1인 파워 유저(본인이 1차 대상). 확장 시 에이전트를 운영하는 소규모 개발팀.
- **사용 환경:** macOS 로컬. Claude Code 세션 JSONL(`~/.claude/projects/**`)을 read-only로 감시. 백그라운드 데몬(launchd) + CLI.
- **사용자 문제:** 에이전트를 오래 돌릴수록 헛돌기·가짜 성공을 놓쳐 토큰·시간을 낭비하고, 최악의 경우 잘못된 "완료"를 신뢰해 사고로 이어진다.
- **사용 목적:** 에이전트를 떠나 있어도 의미적 실패가 발생하면 근거와 함께 즉시 호출받아, 사람이 개입할 시점을 놓치지 않기 위함.

### 3-2 서비스 사용 시나리오

| 단계 | 사용자 행동 | 시스템 동작 |
|---|---|---|
| 1 | `loopbreaker start`로 데몬 기동 후 에이전트 작업 시작 | lockfile로 단일 인스턴스 보장, 세션 JSONL watch 시작 |
| 2 | 에이전트가 같은 파일을 미세 변형하며 반복 편집(헛돌기) | 구조 게이트가 슬라이딩 윈도 반복(Jaccard≥0.3)을 후보로 포착 |
| 3 | (의미 판정 단계) | 게이트 통과분만 임베딩 유사도+LLM-judge로 의미 확인, DetectionRecord 누적 |
| 4 | 데스크톱 알림 수신("thrashing 의심: 같은 영역 9회 재편집, 근거 이벤트 N개") | VerdictRouter가 임계+디바운스 통과 시 채널로 발송 |
| 5 | 사용자가 개입(에이전트 중단/지시 수정) 또는 무시 | 쿨다운으로 동일 신호 재알림 억제 |

### 3-3 주요 기능

| 기능 이름 | 기능 설명 |
|---|---|
| 세션 인제스트(M1) | JSONL 라인→정규화 이벤트, ts/parentUuid/byteOffset 정렬, TailReader 증분 read, 멱등 적재 |
| 구조 게이트(M2) | ActionTriple 생성, 슬라이딩 윈도 동일파일 반복 편집 탐지(미세변형=Jaccard 유사도) |
| 의미 판정(M3) | 임베딩 코사인 유사도 + LLM-as-judge(루브릭·self-consistency·position-swap), fail-closed |
| 알림·사람 호출(M4) | VerdictRouter(임계+디바운스), 채널 sink 4종(Desktop/Webhook/CLI/Mock), 쿨다운 |
| 데몬(M5) | 단일 메인루프 start/stop, 세션별 직렬 파이프라인, WAL 단일 writer, gracefulShutdown |
| 평가 하니스(M6) | 골드셋 라벨, replay(동일 파이프라인 재구동), metrics/calibrate/report |

## 4. 데이터 및 기술 분석

### 4-1 데이터셋 설명

- **데이터 이름:** LoopBreaker 골드셋(.goldset/goldset.eval.db)
- **데이터 출처:** 본인의 실 Claude Code 세션 로그(`~/.claude/projects/**/*.jsonl`) + dohyun 하니스 자기승인 우회 기록(합성 적응).
- **데이터 규모:** 실 세션 1,845개 마이닝 → thrashing 후보 65건(고유 윈도 dedup 31건). 최종 골드셋 **72건** = thrashing 31(실 JSONL) + none 36(실 JSONL) + false_success 5(dohyun 합성).
- **데이터 특징:** 클래스 불균형(thrashing 43%·none 50%·false_success 7%). 라벨은 휴리스틱 약라벨(`auto_heuristic`, fileEditN≥8 강positive)이며 사람 라벨이 아님 — 정직성 명시. false_success는 실 JSONL에 양성이 없어(0건) dohyun 로그 기반 합성 5건으로만 구성.

### 4-2 데이터 전처리

- **결측치 처리 방법:** 파싱 실패 라인은 건너뛰고 전체 중단 금지(SPEC §5(f)). sessionId가 비면 파일명(.jsonl 제거)으로 폴백.
- **데이터 정규화 방법:** 라인→NormalizedEvent(top-level type, tool/input 추출), ts→parentUuid 위상→byteOffset 순 정렬. 평가 매칭 시 **gold span(연속 전체 이벤트)을 pred window와 동일한 구조신호(file_edit) 단위로 정규화**(아래 6장 핵심 개선).
- **데이터 분할 방법:** 캘리브레이션은 stratified k-fold(k=5), 클래스 비율 보존. 소표본(클래스당 <30)이면 grid 축소 + 정성폴백.
- **데이터 증강 / Feature Engineering:** ActionTriple(actor/verb/object) 추출, 슬라이딩 윈도 반복 카운트(structuralRepeatCount), 동일 영역 판정 Jaccard 유사도, false_success는 immediateApproval(Δts≈0)·circularRef(순환참조) 패턴 합성.

## 5. 모델 설계

- **사용한 알고리즘:** 2단계 하이브리드 탐지. ① 구조 게이트(룰 기반: 슬라이딩 윈도 반복 편집 + Jaccard 미세변형 판정) ② 의미 판정(임베딩 코사인 유사도 + LLM-as-judge). 평가 매칭은 thrashing=IoU span(τ=0.5), false_success=anchor±k(k=5).
- **모델 구조:** events → 구조 게이트(DetectionHit) → hits→triples bridge → 의미판정 STAGE2(embed) → judge(루브릭·CoT·self-consistency·position-swap·majority-vote) → DetectionRecord 단조 누적(gate→embed→judge→final) → VerdictRouter → 알림.
- **주요 하이퍼파라미터:** `fileEditWarn/fileEditCrit`(반복 임계), `simThresh=0.9`(임베딩 유사도 임계), Jaccard 0.3(미세변형), `decideThresh`(알림 신뢰도 임계), 디바운스 쿨다운, judge self-consistency 샘플 수. 평가: IoU τ=0.5, anchor k=5, minSupport=15, λ=0.5(FPR_none 페널티).
- **학습 방법:** 별도 가중치 학습이 아니라 **임계 캘리브레이션**(stratified k-fold 격자 탐색, objective=macroF1−λ·FPR_none). 임베딩·judge는 외부 API(Voyage/OpenAI·Anthropic) 또는 결정론 Mock. 본 평가는 Mock judge(결정론) 사용.

## 6. 모델 성능 분석

### 6-1 성능 평가 지표

| 평가 지표 | 설명 |
|---|---|
| Precision/Recall/F1 | 클래스별 정밀도·재현율·조화평균. 주 지표. |
| Macro F1 | 클래스별 F1 단순 평균(skipped 제외). 클래스 불균형 보정 주 지표. |
| Micro F1 | 전체 TP/FP/FN 합산 F1. |
| Balanced Accuracy | 클래스별 recall 평균. 불균형 보정. |
| Cohen's κ | 라벨-예측 일치도(우연 보정). 소표본이면 생략. |
| Wilson 95% CI | 모든 점추정에 신뢰구간 동반(소표본 검정력 보정). |

> RMSE는 회귀 지표라 본 분류 과제엔 미적용. 양식의 RMSE 항목은 해당 없음(분류 과제).

### 6-2 Baseline 모델 성능

Baseline = 평가 매칭에서 **gold span(연속 전체 이벤트)과 pred window(구조 게이트 file_edit만)의 단위 불일치**를 보정하지 않은 상태. 동일 구간을 가리켜도 IoU가 |file_edit|/|전체이벤트| ≈ 0.11로 떨어져 IoU τ=0.5를 못 넘는다.

| 클래스 | 평가 지표 | 성능 |
|---|---|---|
| thrashing | Precision / Recall / F1 | 0.000 / 0.000 / 0.000 (TP=0, FP=59, FN=31) |
| none | Precision / Recall / F1 | 0.500 / 0.379 / 0.431 |
| false_success | F1 | 정성폴백(support 5 < 15) |
| 전체 | Macro F1 / Micro F1 | 0.216 / 0.275 |

### 6-3 성능 개선 실험

| 실험 | 모델/변경 | 성능(Macro F1) |
|---|---|---|
| Baseline | gold span = 전체 이벤트 슬라이스로 IoU 계산 | 0.216 (thrashing F1 0.000) |
| Experiment 1 | gold span을 pred 단위(file_edit)로 정규화(structuralUuids = pred windowRefs 합집합) | thrashing 매칭 회복, 부분겹침 과탐 위험 점검 |
| Experiment 2 | structuralUuids에 gold span 양끝(start/end) 포함 → 부분겹침 시 union 잔류로 FN 유지(과탐 방지) | **0.619 (thrashing F1 0.600)** |
| Experiment 3 | 캘리브레이션 격자(simThresh/fileEditCrit/decideThresh) k-fold 탐색 | 소표본 정성폴백(grid 축소), 주 지표 불변 |

### 6-4 최종 모델 성능

| 항목 | 내용 |
|---|---|
| 최종 모델 | 2단계 하이브리드 탐지 + 단위 정규화 IoU 매칭(Experiment 2 적용) |
| 최종 성능 | Macro F1 **0.619**, Micro F1 0.606, Balanced Accuracy 0.700. thrashing Recall **0.871**(27/31, Wilson 95% CI 0.71~0.95)·Precision 0.458·F1 0.600. none P 0.800/R 0.529/F1 0.637. false_success는 support 5<15 정성폴백. |
| 적용 기술 | IoU span 매칭 단위 정규화, Wilson CI, stratified k-fold 캘리브레이션, replay 결정론 재구동, Mock judge(부수효과 0) |

### 6-5 모델 평가 분석

| 항목 | 내용 |
|---|---|
| Train 성능 | 별도 가중치 학습 없음(룰+임계 모델). 캘리브레이션 k-fold 내부 폴드 macroF1로 대체 측정 → 소표본 정성폴백. |
| Test 성능 | 골드셋 72건 전체 replay: Macro F1 0.619, thrashing Recall 0.871/F1 0.600. |
| 성능 차이 분석 | 정규화 전→후 thrashing F1 0.000→0.600, Macro F1 0.216→0.619로 약 2.9배 개선. 차이의 원인은 학습 부족이 아니라 **평가 매칭의 단위 불일치 버그**였고, 이를 통합 평가가 비로소 드러냈다. |
| 주요 오류 사례 | ① thrashing FP 32건: 구조 게이트가 같은 세션 내 여러 겹치는 윈도를 발화 → 골드 1건당 pred 여러 개 매칭 후 잉여가 FP. ② thrashing FN 4건: 골드 span 양끝이 pred 윈도와 어긋난 소수 케이스. ③ false_success 5건 전부 미발화(FN): 합성 라벨만 있고 대응 JSONL 없음 → 의미판정 입력 부재. |
| 오류 발생 원인 | FP=구조 게이트의 다중 윈도 발화 특성(중복 후보), FN(false_success)=합성 데이터 한계(실 JSONL 양성 0건). 둘 다 설계상 알려진 한계이며 리포트에 정직하게 명시. |

### 6-6 추가 검증 방법

| 항목 | 내용 |
|---|---|
| 교차 검증 | calibrate가 stratified k-fold(k=5)로 임계 격자 탐색. 소표본이라 grid 2~3점 축소 + nested-CV 외부폴드만 일반화 추정(과적합 경고 동반). |
| 추가 테스트 데이터 | 데모는 라이브 + 녹화 JSONL 리플레이(결정론). 실 judge 정량치는 ANTHROPIC_API_KEY+record 모드로 별도 측정 가능(후속). |
| 반복 실험 결과 | 동일 입력→동일 결과(결정론) 단위 테스트로 검증. 전체 회귀: 229 suites / 4,707 tests 통과, 회귀 0. |

## 7. 시스템 구조

```
[Claude Code 세션 JSONL]
        │ (read-only watch, chokidar)
        ▼
[데몬 메인루프 (M5)] ── lockfile 단일 인스턴스
        │ 세션별 직렬 파이프라인
        ▼
[인제스트 M1] 정규화·정렬·멱등 적재 (SQLite op.db, WAL)
        ▼
[구조 게이트 M2] 슬라이딩 윈도 반복 편집 탐지 → DetectionHit
        │ (게이트 통과분만)
        ▼
[의미 판정 M3] 임베딩 유사도 + LLM-as-judge → DetectionRecord
        ▼
[VerdictRouter M4] 임계+디바운스 → NotifyDispatcher
        ▼
[알림 채널] Desktop / Webhook / CLI  →  사람 호출
        
[평가 하니스 M6] 골드셋 → replay(동일 파이프라인) → metrics/calibrate/report (eval.db)
```

사용자 → CLI/데몬 → 세션 watch → 2단계 탐지(구조→의미) → 알림 → 사람 개입. 평가 하니스는 같은 파이프라인을 녹화 JSONL로 재구동해 메트릭 산출.

## 8. 프로젝트 수행 과정

### 8-1 개발 일정

| 주차/회차 | 수행 내용 |
|---|---|
| 1차 | 주제 발굴·확정, 사전조사 기획서(8항목), 개발명세 SPEC(5영역+contracts SSOT+정합화 패치노트), 골격 M0 |
| 2차 | 인제스트 M1(파서·정렬·tail·멱등), 구조 게이트 M2(반복 편집·Jaccard) |
| 3차 | 의미 판정 M3(임베딩+judge), 알림 M4(라우터·sink), 데몬 통합 M5(메인루프) |
| 4차 | 평가 하니스 M6(metrics/calibrate/replay/report), 제품 포장(CLI·launchd), 골드셋 마이닝, 실측 평가 |

### 8-2 프로젝트 진행 과정

- **데이터 수집:** 실 Claude Code 세션 1,845개를 MANUAL-ONLY CLI로 마이닝(개인정보라 평가 본체와 분리). 구조 신호 후보 65건 추출 → 고유 윈도 dedup 31건.
- **데이터 전처리:** 후보를 약라벨(휴리스틱)로 thrashing 31건 라벨링, 정상 세션에서 none anchor 36건 샘플링, dohyun 자기승인 우회 5건 합성. 골드셋 72건을 eval DB에 적재.
- **모델 개발:** ralph(ouroboros) 자율 코딩으로 M0~M6 구현. 각 마일스톤은 격리 구현 후 통합 글루·경계 정합·통합 테스트를 직접 보완(ralph가 통합부를 timeout으로 놓치는 패턴 대응).
- **성능 개선:** 골드셋으로 실측 리포트 산출 중 **thrashing이 전부 FN(F1 0)** 나오는 현상 발견 → 원인이 gold span↔pred window 단위 불일치임을 통합 평가로 규명 → IoU 정규화로 수정 → Macro F1 0.216→0.619.

## 9. 프로젝트 결과

- **개발된 시스템:** LoopBreaker 전체(M0~M6) + 제품 포장(`loopbreaker` CLI start/stop/status/doctor, 데몬 엔트리, launchd plist). 데몬(운영 경로)과 평가 하니스(검증 경로) 양면 동작. 코드 규모 src 약 18,600 LOC, tests 약 66,800 LOC.
- **최종 모델 성능:** Macro F1 0.619, thrashing Recall 0.871(CI 0.71~0.95)/F1 0.600, none F1 0.637. false_success 정성폴백. 229 suites/4,707 tests 통과.
- **해결된 문제:** 규칙으로 못 잡는 thrashing(미세변형 반복)을 구조 신호로 높은 재현율(0.871)로 탐지. 가짜 성공은 패턴 정의·합성 시드까지 확보(실 양성 데이터 확대는 후속).
- **사용자에게 제공되는 가치:** 에이전트를 떠나 있어도 헛도는 순간 근거와 함께 호출받아, 토큰·시간 낭비와 잘못된 "완료" 신뢰로 인한 사고를 조기에 차단할 수 있다.

## 10. 프로젝트 한계 및 개선 방향

**프로젝트 한계** (SPEC §8 6대 한계, 리포트에 정직하게 동반)

1. 참조 상한: IBM 하이브리드 사이클 탐지 F1=0.72(arXiv:2511.10650)는 도메인·골드셋이 달라 목표가 아니라 참고점일 뿐. 초과를 주장하지 않음.
2. 골드셋 규모: n=72, 단일 라벨러(본인). 검정력이 낮아 모든 지표에 Wilson 95% CI 동반, support 작은 클래스(false_success)는 CI가 넓음.
3. 라벨 신뢰 상한: intra-rater κ가 모델 성능의 천장. 모델 κ ≈ 라벨 κ면 "사람만큼 일관적"이지 그 이상이 아님.
4. 합성 분리: synthetic(false_success) 양성으로 부풀린 수치를 주 지표로 쓰지 않음. 실데이터-only가 결론.
5. 캘리브레이션 과적합: 임계는 이 골드셋에 fit. 새 프로젝트/스타일에서 재캘리브 필요. nested-CV 외부 폴드만 일반화 추정.
6. 정량 목표 미설정: "정밀도 X% 달성"식 약속을 박지 않음. "이 방법으로 측정했고, 측정값은 이렇고, 이 한계 안에서 해석하라"가 정량 증거.

**향후 개선방향**

- false_success 실 양성 데이터 확보(다중 라벨러 라벨링 → intra-rater κ 천장 상향).
- 실 Anthropic judge API record 모드로 의미판정 정량치 측정(현재 Mock).
- 자동 개입(PreToolUse deny·Stop block)으로 "Detect first, Block later" 완성.
- Windows/Linux 확장, nested cross-validation으로 캘리브레이션 과적합 방지.

## 11. 프로젝트 회고 (KPT)

**Keep — 잘된 점**

- 문제 정의를 가설이 아니라 내 실제 세션 로그 실측(자기승인 우회 5건)으로 잡아 설득력 확보.
- 정직성 유지: 정량 목표를 약속하지 않고 모든 지표에 Wilson CI, 소표본은 정성폴백, 합성은 부록.
- 부수효과 격리: 평가도 Mock·임시경로·합성 픽스처만으로 테스트(부수효과 0), 실 마이닝은 MANUAL-ONLY 분리.
- "만든 만큼 측정한다" — 탐지 코드만큼 평가 코드를 크게 짜고, 통합 평가로 진짜 성능을 드러냄.

**Problem — 어려웠던 점**

- 자율 코딩 에이전트(ralph)가 부품은 잘 만들지만 통합 글루·마일스톤 경계·조립부(daemon 메인루프, calibrate/report)를 timeout으로 놓침.
- 실버그가 통합 단계에서야 드러남: ① M6 op_main read-only가 eval 본체 쓰기까지 막음 ② **실측 평가에서 gold span↔pred window 단위 불일치로 thrashing이 전부 FN**.

**Try — 다음 프로젝트에서 시도할 것**

- 마일스톤 머지 후 반드시 end-to-end 통합 테스트로 경계 검증을 루틴화(이번에 단위 테스트가 놓친 단위 불일치를 통합 평가가 잡음).
- ralph가 놓치는 조립부는 부품 시그니처 수집 → 설계서 → 글루 직접 작성 절차로 표준화.
- 평가 매칭에서 gold/pred의 단위(전체 이벤트 vs 구조신호)가 같은지 설계 시점에 명시적으로 못 박기.
