# 모델 성능 최적화 포트폴리오 (2차) — LoopBreaker

> 성능 최적화 모델 구현 및 개선 분석. 모든 성능은 실제 실험(실 세션 골드셋 72건 리플레이) 결과 기반. Baseline(1차) 대비 개선 포함.

## 1. 기본 정보

- **프로젝트 제목:** LoopBreaker — 코딩 AI 에이전트의 의미적 실패(헛돌기·가짜 성공) 탐지 모델
- **학번:** 3413
- **이름:** 지도현

## 2. 문제 정의 및 예측 목표

- **해결하려는 문제:** 자율 AI 코딩 에이전트의 의미적 실패(① thrashing=미세변형 반복 편집 헛돌기 ② false_success=근거 없는 가짜 완료 선언)를 자동 탐지. (1차와 동일 문제.)
- **문제 발생 배경:** 결정론적 규칙으로는 안 잡히는 의미적 실패. 1차 Baseline에서 모델이 실제로는 thrashing 후보를 잘 포착함에도 **평가 매칭 단위 불일치**로 thrashing F1=0이 나오는 한계를 확인 → 이를 최적화하는 것이 2차 목표.
- **모델의 예측 목표:** 세션 로그 윈도가 thrashing / false_success / none 중 어디인지 다중 클래스 분류. (동일.)
- **입력 데이터:** 정규화·정렬된 이벤트 시퀀스(ActionTriple) + 슬라이딩 윈도 구조 신호. **(2차: 평가 매칭 시 gold span을 pred 윈도와 동일한 file_edit 단위로 정규화한 입력 사용.)**
- **출력 결과:** 탐지 라벨(thrashing/false_success/none) + 근거. 골드 라벨과 단위 정합 매칭으로 혼동행렬 산출.

## 3. 데이터셋 설명

- **데이터 이름:** LoopBreaker 골드셋(`.goldset/goldset.eval.db`) — 1차와 동일 데이터(개선은 모델/평가 쪽, 데이터 누수 없음).
- **데이터 출처:** 실 Claude Code 세션 로그 1,845개 마이닝 + dohyun 자기승인 우회 합성.
- **데이터 개수(전체 / Train / Test):** 전체 **72건**. 캘리브레이션 stratified k-fold(k=5) 내부 분할(소표본 → grid 축소 + nested-CV 외부폴드만 일반화 추정). 최종 성능은 전체 72건 리플레이(Test 전수).
- **데이터 특징:** thrashing 31·none 36·false_success 5. 클래스 불균형(43%/50%/7%). 휴리스틱 약라벨, false_success는 합성 소표본.

## 4. 데이터 전처리 과정

- **결측치 처리 방법:** 파싱 실패 라인 skip(전체 중단 금지), sessionId 빈 값은 파일명 폴백. (1차와 동일.)
- **데이터 정규화 방법(★ 2차 핵심 개선):** 평가 매칭에서 **gold span(연속 전체 이벤트)을 pred 윈도와 동일한 구조신호(file_edit) 단위로 정규화.** `matchThrashing`에 `structuralUuids`(= 세션 내 thrashing pred windowRefs 합집합 ∪ gold span 양끝)를 도입해, gold span 슬라이스를 이 집합과 교집합 → file_edit 단위로 축소 후 IoU 계산. 동일 구간 IoU가 ≈0.11 → ≈1.0으로 회복.
- **데이터 분할 방법:** stratified k-fold(k=5). 소표본이면 grid 2~3점 축소 + 정성폴백.
- **데이터 증강 / Feature Engineering:** ActionTriple, structuralRepeatCount, Jaccard 유사도, false_success 합성 패턴(immediateApproval·circularRef). (1차와 동일.)

## 5. Baseline 모델 설계 (최적화 모델 포함)

- **사용 알고리즘:** 2단계 하이브리드 탐지(구조 게이트 + 의미 판정) + **단위 정규화 IoU 매칭**. thrashing=IoU span(τ=0.5, 정규화), false_success=anchor±k(k=5).
- **모델 구조:** events → 구조 게이트 → hits→triples → 의미판정(embed) → judge → DetectionRecord. **평가 매칭부에 structuralUuids 정규화 적용** + **stratified k-fold 임계 캘리브레이션**(objective=macroF1−λ·FPR_none, λ=0.5, DEFAULT copy+override로 원본 불변).
- **주요 하이퍼파라미터:** simThresh=0.9, fileEditCrit/Warn, Jaccard 0.3, 평가 IoU τ=0.5, anchor k=5, minSupport=15, λ=0.5. 캘리브레이션 candidate 격자: {기본}, {simThresh 0.85}, {CRITICAL 15}, {decideThresh 0.6}.
- **학습 방법:** 가중치 학습 없이 임계 캘리브레이션. 임베딩·judge는 결정론 Mock. 동일 입력→동일 결과(결정론) 보장.

## 6. 성능 평가 방법

| 평가 지표 | 사용 여부 | 설명 |
|---|---|---|
| Accuracy | △ (부록) | 클래스 불균형이라 주 지표 아님. 최종 0.606. |
| F1-score | ✅ 사용 | 클래스별 F1 + Macro F1(주 지표) + Micro F1. |
| RMSE | ❌ 미사용 | 회귀 지표 — 분류 과제 해당 없음. |

> 추가: Precision/Recall(클래스별, Wilson 95% CI 동반), Balanced Accuracy, Cohen's κ. false_success(support 5<15)는 정성폴백.

## 7. Baseline 모델 성능 결과 (최적화 전후 비교)

| 실험 | 모델 | 평가지표 | 성능 |
|---|---|---|---|
| Baseline | 정규화 미적용(1차) | Macro F1 / thrashing F1 | 0.216 / 0.000 |
| **최적화 모델** | **단위 정규화 IoU 적용(2차)** | **Macro F1 / thrashing F1** | **0.619 / 0.600** |
| 최적화 모델 | 〃 | thrashing P/R/F1 | 0.458 / **0.871** (CI 0.71~0.95) / 0.600 (TP=27, FP=32, FN=4) |
| 최적화 모델 | 〃 | none P/R/F1 | 0.800 / 0.529 / 0.637 |
| 최적화 모델 | 〃 | Micro F1 / Balanced Acc | 0.606 / 0.700 |
| 최적화 모델 | 〃 | false_success | 정성폴백(support 5 < 15) |

> 개선 폭: Macro F1 0.216 → 0.619 (약 2.9배), thrashing F1 0.000 → 0.600, none F1 0.431 → 0.637.

## 8. 초기 모델 분석 (최적화 결과 분석)

- **모델의 장점:** thrashing을 **재현율 0.871**(31건 중 27건 포착, 4건만 누락)로 잡아낸다 — 헛돌기 탐지라는 핵심 목표에서 높은 민감도. 단위 정규화로 같은 구간을 정확히 매칭하면서도 부분겹침은 FN으로 유지해 과탐을 막는다. 모든 지표에 Wilson CI를 달아 신뢰구간을 정직하게 표기.
- **모델의 한계:** ① thrashing Precision 0.458(FP=32) — 구조 게이트가 같은 세션 내 겹치는 윈도를 다중 발화해 잉여 예측이 FP로 쌓임. ② false_success는 실 양성 0건이라 합성 5건 정성폴백(정량 F1/κ 생략). ③ 캘리브레이션은 소표본이라 candidate 간 차이가 작아(정성폴백) 임계 최적화 효과는 제한적. ④ 단일 라벨러·휴리스틱 약라벨이 모델 성능의 천장(intra-rater κ).
- **성능이 낮은 원인 분석:** 남은 약점은 thrashing Precision과 none Recall(0.529)이다. 원인은 구조 게이트의 **다중 윈도 발화 특성** — 하나의 헛돌기 구간에서 길이가 점증하는 여러 윈도를 동시에 발화해, 골드 1건에 매칭되고 남은 윈도가 FP가 된다. none Recall이 낮은 것도 이 FP가 none을 침범하기 때문.
- **개선이 필요한 부분:** ① 겹치는 thrashing 윈도를 NMS(non-maximum suppression)식으로 dedup해 FP 억제 → Precision 향상. ② false_success 실 양성 확보. ③ 다중 라벨러로 라벨 천장 상향.

## 9. 향후 성능 개선 계획

다음 단계에서 시도할 방법:

- **모델 구조 변경:** thrashing 예측 윈도를 NMS/겹침 병합으로 후처리해 골드 1건당 1 예측으로 정규화 → FP 32건 감소 기대(Precision↑, none Recall↑).
- **하이퍼파라미터 조정:** fileEditCrit·decideThresh를 nested cross-validation으로 재탐색(현재 소표본 정성폴백 → 데이터 확대 후 정량 최적화).
- **데이터 증강:** false_success 합성 패턴 확대 + 실 세션에서 자기승인 우회 마이닝 규칙 추가, 다중 라벨러 라벨링으로 골드셋 규모·신뢰도 상향.
- **Feature Engineering:** 의미판정에 실 Anthropic judge API(record 모드) 도입해 thrashing/false_success 의미 신호 정량치 측정(현재 Mock).
- **추가 검증:** 다른 프로젝트/스타일 세션으로 외부 검증(캘리브레이션 일반화 확인), thrashing Precision-Recall 곡선으로 운영 임계 선택.
